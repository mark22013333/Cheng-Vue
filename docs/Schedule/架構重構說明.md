# 任務類型參數定義架構重構說明

## 🎯 重構目標

將參數定義從 Provider 移到 Enum 中，讓每個任務類型自己管理自己的參數配置。

---

## 📊 架構對比

### 重構前

```
CrawlerTypeProvider
├─ buildParams(type) {
│    switch (type) {
│      case CA101: return buildCA101Params();
│      case CA102: return buildCA102Params();
│      case CA103: return buildCA103Params();
│    }
│  }
├─ buildCA101Params() { ... }
├─ buildCA102Params() { ... }
└─ buildCA103Params() { ... }
```

**問題：**
- ❌ 新增爬蟲類型需要修改兩個地方（enum + provider）
- ❌ 參數定義和類型定義分離，容易遺漏
- ❌ Provider 程式碼冗長，每個類型都要一個方法

### 重構後

```
CrawlerType (enum)
├─ CA101 { buildParams() { ... } }
├─ CA102 { buildParams() { ... } }
└─ CA103 { buildParams() { ... } }
    ↓
CrawlerTypeProvider
└─ buildTaskTypeOption(type) {
     params: type.buildParams() ✨
   }
```

**優點：**
- ✅ 新增爬蟲類型只需修改 enum
- ✅ 參數定義和類型定義在同一處
- ✅ Provider 非常簡潔，不需要 switch-case
- ✅ 強制每個常量實作 buildParams()

## 🔄 重構內容

### 1. 爬蟲模組（cheng-crawler）

#### 修改檔案

**`CrawlerType.java`**
- ✅ 移除 `@AllArgsConstructor`（因為需要自訂建構子）
- ✅ 新增 `abstract List<TaskParamMetadata> buildParams()`
- ✅ 每個常量實作自己的 `buildParams()` 方法
- ✅ CA101、CA102、CA103 的參數定義都移到 enum 中

**`CrawlerTypeProvider.java`**
- ✅ 刪除 `buildParams(CrawlerType)` switch-case
- ✅ 刪除 `buildCA101Params()`, `buildCA102Params()`, `buildCA103Params()`
- ✅ 刪除 `buildCommonParams()`
- ✅ 簡化 `buildTaskTypeOption()` 方法，改用 `type.buildParams()`

### 2. 排程模組（cheng-quartz）

#### 新增檔案

**`NotificationType.java`**
- ✅ 建立推播類型 enum
- ✅ LINE_PUSH 常量及其參數定義

#### 修改檔案

**`NotificationTypeProvider.java`**
- ✅ 刪除 `buildLinePushOption()`, `buildLinePushParams()`
- ✅ 改用 `Arrays.stream(NotificationType.values()).map(...)`
- ✅ 簡化為與 CrawlerTypeProvider 相同的模式

## 📝 使用範例

### 新增一個爬蟲類型

**只需修改一個檔案：`CrawlerType.java`**

```java
public enum CrawlerType {
    CA101("ISBN書籍查詢", "書籍庫存") {
        @Override
        public List<TaskParamMetadata> buildParams() {
            return Collections.singletonList(
                TaskParamMetadata.builder()
                    .name("isbn")
                    .label("ISBN")
                    .type(TaskParamType.STRING)
                    .required(true)
                    .build()
            );
        }
    },
    
    // ✅ 新增：只需在這裡加一個常量
    CA104("櫃買中心", "上櫃公司每日收盤價") {
        @Override
        public List<TaskParamMetadata> buildParams() {
            return Arrays.asList(
                TaskParamMetadata.builder()
                    .name("mode")
                    .label("執行模式")
                    .type(TaskParamType.SELECT)
                    .required(true)
                    .defaultValue("today-only")
                    .options(...)
                    .build()
            );
        }
    };
    
    // ... 建構子和抽象方法定義
}
```

**CrawlerTypeProvider 完全不需要修改！** 🎉

## 📐 設計模式

### 使用的設計模式

1. **策略模式（Strategy Pattern）**
   - 每個 enum 常量是一個策略
   - `buildParams()` 是策略方法

2. **模板方法模式（Template Method）**
   - TaskTypeProvider 定義統一的處理流程
   - Enum 實作具體的參數定義

3. **工廠模式（Factory Pattern）**
   - TaskTypeProvider 作為工廠
   - 產生 TaskTypeOption 物件

### 符合的設計原則

- ✅ **單一職責原則（SRP）**：每個 enum 常量只負責自己的參數
- ✅ **開放封閉原則（OCP）**：新增類型不需修改 Provider
- ✅ **依賴反轉原則（DIP）**：Provider 依賴抽象方法，不依賴具體實作

}

## 💡 總結

這次重構大幅簡化了程式碼結構，讓每個任務類型自己管理參數定義。未來新增任務類型時，只需要：

1. 在對應的 enum 中新增常量
2. 實作 buildParams() 方法
3. 實作對應的 Handler/Task

**Provider 完全不需要修改！** 🎉

這種架構更符合物件導向設計原則，也更容易維護和擴展。
