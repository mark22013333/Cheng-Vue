## 🏗️ 系統架構說明

### 核心概念

```
┌─────────────────────────────────────────────────────────┐
│  各模組提供任務類型                                        │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │ 爬蟲模組      │  │ 推播模組      │  │ 報表模組      │  │
│  │ CrawlerType  │  │ Notification │  │ ReportType   │  │
│  │ Provider     │  │ Provider     │  │ Provider     │  │
│  └──────┬───────┘  └──────┬───────┘  └──────┬───────┘  │
│         │                  │                  │          │
│         └──────────────────┼──────────────────┘          │
│                            ↓                             │
│              ┌─────────────────────────┐                 │
│              │   cheng-quartz 模組      │                 │
│              │  自動收集所有 Provider   │                 │
│              │  提供統一 API 給前端     │                 │
│              └────────────┬────────────┘                 │
│                           ↓                              │
│              ┌─────────────────────────┐                 │
│              │  API: /monitor/job/     │                 │
│              │       taskTypes         │                 │
│              └────────────┬────────────┘                 │
└───────────────────────────┼──────────────────────────────┘
                            ↓
              ┌─────────────────────────┐
              │      前端頁面            │
              │  /monitor/job           │
              │  - 取得任務類型列表      │
              │  - 動態產生參數表單      │
              │  - 新增/編輯排程         │
              └─────────────────────────┘
```

### 資料流向

1. **啟動時**：Spring 自動掃描所有 `TaskTypeProvider` 實作
2. **前端請求**：呼叫 `/monitor/job/taskTypes` 取得所有可用的任務類型
3. **使用者操作**：選擇任務類型後，前端根據參數定義動態產生表單
4. **儲存排程**：將任務類型代碼和參數組合成 `invokeTarget`
5. **執行排程**：Quartz 呼叫對應的 Bean 方法，傳入參數

#### 步驟 2：建立 TaskTypeProvider

```java
// 檔案：cheng-quartz/src/main/java/com/cheng/quartz/provider/ReportTypeProvider.java

@Slf4j
@Component
public class ReportTypeProvider implements TaskTypeProvider {
    
    @Override
    public TaskCategory getCategory() {
        return TaskCategory.REPORT;
    }
    
    @Override
    public List<TaskTypeOption> getTaskTypes() {
        return Arrays.stream(ReportType.values())
                .map(this::buildTaskTypeOption)
                .collect(Collectors.toList());
    }
    
    @Override
    public int getPriority() {
        return 30; // 報表任務優先級
    }
    
    private TaskTypeOption buildTaskTypeOption(ReportType type) {
        return TaskTypeOption.builder()
                .code(type.name())
                .label(String.format("[%s] %s - %s",
                        type.name(),
                        type.getCategory(),
                        type.getFuncName()))
                .category(TaskCategory.REPORT.getCode())
                .description(String.format("產生「%s」類型的「%s」",
                        type.getCategory(),
                        type.getFuncName()))
                .beanName("reportTask")
                .defaultMethod("generateReport")
                .params(type.buildParams()) // ✨ 直接呼叫 enum 的方法
                .enabled(true)
                .build();
    }
}
```

## 📐 架構優勢總結

### 🎯 重構後的架構

```
每個任務類型自己管理參數
┌─────────────────────────────────┐
│   CrawlerType (enum)            │
│   ├─ CA101 + buildParams()      │
│   ├─ CA102 + buildParams()      │
│   └─ CA103 + buildParams()      │
└─────────────────────────────────┘
          ↓ 直接呼叫
┌─────────────────────────────────┐
│   CrawlerTypeProvider           │
│   只負責包裝成 TaskTypeOption   │
│   非常簡潔！                     │
└─────────────────────────────────┘
```

### ✅ 優點

1. **責任清晰**：每個 enum 常量負責自己的參數定義
2. **易於維護**：新增爬蟲類型只需修改 enum，Provider 完全不需要動
3. **類型安全**：參數定義和類型定義在同一個地方，不會遺漏
4. **擴展性強**：新增任務分類也是相同模式，一致性高
5. **符合 OO 原則**：封裝、單一職責原則

### 📊 對比表

| 項目 | 舊架構 | 新架構 |
|------|--------|--------|
| 新增爬蟲類型 | 修改 enum + 修改 Provider | 只修改 enum |
| 參數定義位置 | Provider 的 switch-case | Enum 的常量實作 |
| 程式碼重複 | 高（每個參數定義一個方法） | 低（直接在常量中定義） |
| 維護難度 | 中等（容易漏改） | 低（強制實作） |
| 擴展性 | 中等 | 高 |

## ❓ 常見問題

### Q1: 為什麼我的 API 回傳空列表？

**A:** 檢查以下幾點：

1. **確認 TaskTypeProvider 是否被 Spring 掃描到**

```java
// 確認類別有 @Component 註解
@Component
public class CrawlerTypeProvider implements TaskTypeProvider {
    // ...
}
```

2. **確認模組是否正確載入**

```bash
# 查看啟動日誌，應該會看到類似：
# CrawlerHandler Initialized: CA103WHandler
# 如果沒有，表示模組沒有被掃描到
```

3. **確認 cheng-crawler 是否被 cheng-admin 依賴**

```xml
<!-- cheng-admin/pom.xml 應該要有 -->
<dependency>
    <groupId>com.cheng</groupId>
    <artifactId>cheng-crawler</artifactId>
</dependency>
```

### Q2: 如何測試單一爬蟲是否正常運作？

**A:** 使用 Quartz 的「立即執行」功能：

1. 在後台新增排程
2. 呼叫目標填寫：`crawlerTask.runWithMode('CA103', 'today-only')`
3. 設定任何 Cron 表達式（因為要手動執行）
4. 儲存後，點擊「執行一次」
5. 查看執行日誌

### Q3: 參數要如何傳遞給 Handler？

**A:** 參數傳遞流程：

```
前端表單 → API 儲存 → invokeTarget
  例如：crawlerTask.runWithMode('CA103', 'today-only')
         ↓
Quartz 執行 → CrawlerTask.runWithMode("CA103", "today-only")
         ↓
建立 CrawlerParams → params.mode = "today-only"
         ↓
Handler 讀取 → String mode = params.getMode()
```

### Q4: 如何新增自訂的參數類型？

**A:** 在 `TaskParamType.java` 新增：

```java
public enum TaskParamType {
    // ... 現有類型
    
    // 新增：時間範圍選擇器
    TIME_RANGE("時間範圍", "timeRange"),
    
    // 新增：檔案上傳
    FILE("檔案", "file");
}
```

然後在前端根據新類型實作對應的輸入組件。
