# 定時任務分類設計文件

## 一、設計理念

採用**兩層分類架構**：

```
第一層：任務分組（jobGroup）- 粗分類，用於快速篩選和管理
第二層：任務類型（taskType）- 細分類，提供具體的範本和參數配置
```

---

## 二、任務分組（jobGroup）- 主分類

| 代碼 | 名稱 | 標籤顏色 | 說明 | 適用場景 |
|------|------|----------|------|----------|
| `DATA` | 資料處理 | primary（藍） | 資料同步、匯入匯出、格式轉換 | ISBN抓取、庫存同步、資料遷移 |
| `MAINTENANCE` | 系統維護 | info（灰） | 備份、清理、健康檢查 | 日誌清理、資料庫備份、快取清理 |
| `NOTIFICATION` | 通知推播 | warning（橙） | LINE、郵件、簡訊 | 每日報表推送、庫存預警、活動通知 |
| `REPORT` | 報表統計 | success（綠） | 日報、週報、月報 | 銷售統計、庫存盤點、使用分析 |
| `BUSINESS` | 業務處理 | danger（紅） | 訂單、支付、對帳 | 自動對帳、訂單同步、狀態更新 |
| `CRAWLER` | 爬蟲抓取 | 預設（無） | 網頁抓取、API呼叫 | 價格監控、資料蒐集、內容更新 |

## 五、使用流程

### 新增任務時的使用體驗

1. **選擇任務分組**（主分類）
   ```
   [ 下拉選單 ]
   ├─ 📊 資料處理
   ├─ 🔧 系統維護
   ├─ 📣 通知推播
   ├─ 📈 報表統計
   ├─ 💼 業務處理
   └─ 🕷️ 爬蟲抓取
   ```

2. **選擇設定方式**
   ```
   ( ) 從範本選擇  ( ) 手動輸入
   ```

3. **從範本選擇任務類型**（子分類，按分組過濾）
   ```
   [ 任務類型下拉選單 ]
   
   資料處理
   ├─ 書籍ISBN重新抓取 - 重新抓取書籍的ISBN資訊
   ├─ 庫存資料同步 - 同步各倉庫的庫存資料
   └─ Excel批次匯入 - 從Excel批次匯入資料
   
   通知推播
   ├─ LINE推播訊息 - 透過LINE推播訊息
   └─ 每日報表郵件 - 發送每日報表郵件
   ```

4. **自動填入建議設定**
   - Cron 表達式自動填入
   - 呼叫方法自動填入
   - 顯示參數表單（如果有）

5. **調整參數**（如果範本有定義參數）
   ```
   任務參數
   ┌─────────────────────────────────┐
   │ 倉庫ID列表（選填）               │
   │ [_______________________]       │
   │ 範例: 1,2,3                     │
   └─────────────────────────────────┘
   ```

## 六、優勢分析

### ✅ 使用兩層分類的好處

1. **快速篩選** - 在列表頁按主分類快速找到相關任務
2. **清晰分類** - 一眼就能看出任務的用途
3. **範本管理** - 同分類的任務有相似的設定方式
4. **權限控制** - 可以按分組設定操作權限（未來擴展）
5. **統計分析** - 可以按分組統計任務執行情況

### 🎯 相比原有 DEFAULT/SYSTEM 的改進

| 項目 | 原有方式 | 改進方式 |
|------|---------|---------|
| 分類數量 | 2個（預設、系統） | 6個業務分類 |
| 語義性 | 不明確 | 清晰明確 |
| 擴展性 | 困難 | 容易新增分類 |
| 範本支援 | 無 | 每個分類有對應範本 |
| 使用者體驗 | 需要手動輸入 | 選擇範本自動填入 |

## 七、後續擴展建議

### 7.1 實現後端任務類型 API

建立 `JobTypeController` 提供以下 API：

```java
// 取得所有任務類型
GET /monitor/job/types

// 根據分組取得任務類型
GET /monitor/job/types/category/{category}

// 根據代碼取得任務類型詳情
GET /monitor/job/types/{code}
```

### 7.2 任務類型可配置化

將任務類型配置儲存到資料庫，支援動態新增、修改、刪除。

### 7.3 權限控制

根據 `jobGroup` 設定操作權限：
- 某些使用者只能管理「資料處理」分組的任務
- 「系統維護」分組的任務只有管理員能操作

### 7.4 執行統計

按分組統計任務執行情況：
- 各分組任務數量
- 各分組成功率
- 各分組平均執行時間

## 八、實施建議

### 階段一：更新字典數據（立即可做）
✅ 執行 `V28__update_job_group.sql` 更新任務分組選項

將現有任務的 `jobGroup` 更新為新的分類

### 階段三：實現任務類型 API（後端開發）
1. 建立 `JobType` 實體類

4. 準備任務類型初始數據

### 階段四：前端整合（前端開發）
前端已經有範本選擇功能，只需：
1. 確保能正常呼叫 `/monitor/job/types` API
2. 測試範本選擇功能
3. 優化 UI/UX

## 九、注意事項

⚠️ **資料遷移**：更新字典後，記得更新現有任務的 `jobGroup` 值

⚠️ **向後兼容**：如果有程式碼硬編碼 `DEFAULT` 或 `SYSTEM`，需要一併修改

⚠️ **Quartz JobKey**：變更 `jobGroup` 會影響 Quartz 的 JobKey，建議在系統空檔時進行
